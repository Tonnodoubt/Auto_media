# 阿里云 ECS + FRP 公网穿透设计方案

> 说明：本文已根据 2026-03-28 对当前仓库代码的实际核查结果更新，结论以“当前代码不修改”为前提。

> 文档状态：设计方案 / 可落地实施说明  
> 日期：2026-03-28  
> 目标：通过阿里云 ECS 的公网 IP 或 EIP 部署 `frps`，在本机部署 `frpc`，让外部用户可以从任意网络访问本机运行的 AutoMedia 服务。

## 快速结论

如果你只想先看最终判断，先看这 5 句：

1. **FRP 公网穿透本身完全可行。**
2. **双端口方案只适合验证链路，不适合直接作为完整公网方案。**
3. **当前不改代码时，最稳的公网方式是“单端口同源入口 + 手动设置同一公网 `backendUrl` + 服务端预置所需 `.env` Key”。**
4. **如果目标是“外部用户不在本机、只打开网址就直接完整使用真实功能”，当前代码和当前文档都不能直接保证。**
5. **如果统一入口仍是 `http://<ECS公网IP>:...`，不要让外部用户在前端设置页填写真实第三方 API Key；优先改成 HTTPS，或只使用服务端 `.env` 默认 Key。**

---

## 1. 需求目标

你要实现的核心效果是：

1. 本机继续作为实际业务服务器运行。
2. 阿里云 ECS 作为中转入口，具备公网访问能力。
3. 外部用户不需要和你在同一局域网，也不需要 VPN。
4. 只要本机服务和 `frpc` 在线，用户就能通过阿里云公网地址访问你的本机服务。

结合当前仓库，现状是：

- 后端默认运行在 `http://localhost:8000`
- 前端开发服务器默认运行在 `http://localhost:5173`
- `frontend/vite.config.js` 已配置 `/api`、`/health`、`/media` 代理到本机 `8000`

这意味着：`frp` 本身可以打通公网访问，但如果要让“整个 Web 应用”对外稳定可用，还需要同时考虑前端暴露方式、后端地址回写、CORS 和媒体资源 URL。

---

## 2. 可行性结论

## 2.1 结论

需要把两个目标分开看：

1. **目标 A：把本机服务通过阿里云 ECS 暴露到公网**
   - 这个目标**完全可行**
   - 而且是典型的 `frp` 使用场景

2. **目标 B：让外部用户“打开网址即零配置完整使用真实功能”**
   - 在**当前代码不修改**前提下，**不能直接保证**
   - 原因不在 `frp`，而在当前仓库的地址回落、浏览器本地设置、Mock 模式和 API Key 策略

只看 FRP 能力的话，方案本身完全成立：

- 阿里云 ECS 提供公网入口。
- 本机不需要公网 IP，也不需要路由器做端口映射。
- `frpc` 主动从本机连到云上的 `frps`，非常适合家庭宽带、公司内网、NAT 环境。

## 2.2 可行性的前提条件

要落地成功，必须满足下面几个条件：

- 阿里云 ECS 必须有公网 IPv4 能力，或者绑定 EIP。
- ECS 安全组必须放行 `frps` 控制端口和对外暴露端口。
- 本机可以主动访问公网，并且到 ECS 的出站连接稳定。
- 本机服务启动后，要么直接暴露对应端口，要么由本机反向代理统一出口。

## 2.3 适用场景

这个方案很适合：

- 内部测试
- 演示环境
- 小范围试用
- 临时公网开放
- 需要保留“服务实际运行在本机”的场景

## 2.4 不适合的场景

如果你追求下面这些目标，这个方案就不是最优解：

- 高并发公网生产服务
- 7x24 小时高可用
- 大量图片/视频分发
- 严格的合规、审计、WAF、DDoS 防护

原因也很直接：

- 所有公网流量都要经过 ECS 中转。
- 本机上行带宽会成为瓶颈。
- 你的项目包含图片、音频、视频文件，带宽和时延会明显影响体验。

## 4. 两种落地方式

## 4.1 方式 A：最快落地的双端口方案

这是**最容易先跑通**的方案，基本不要求先改代码。

### 暴露方式

- ECS `15173` -> 本机 `5173`（前端）
- ECS `18000` -> 本机 `8000`（后端）

### 外部访问效果

- 用户访问前端：`http://<ECS公网IP>:15173`
- 用户访问后端：`http://<ECS公网IP>:18000/docs`

### 适用条件

- 你先要快速验证公网访问链路是否打通。
- 允许前端和后端分别占用两个公网端口。

### 当前仓库下的注意点

这个方案能跑起来，但**不是最优用户体验**，因为当前仓库存在几个现实问题：

1. `app/main.py` 里 CORS 只允许 `http://localhost:5173`。
2. `frontend/src/views/VideoGeneration.vue` 在开发模式下会默认把后端地址当成 `http://localhost:8000`。
3. 某些接口默认参数或文档说明里仍然带有 `localhost:8000`。

所以如果你采用双端口方案，建议额外做三件事：

1. 后端把允许来源补上 `http://<ECS公网IP>:15173`
2. 前端默认后端地址改为 `http://<ECS公网IP>:18000` 或支持环境变量
3. 媒体 URL 和 `base_url` 统一改为公网可达地址

如果坚持“**当前先不改代码**”，那这一套双端口方案就应该只拿来做**FRP 链路验证**，不要直接当成最终给外部用户长期使用的公网方案。

## 4.2 方式 B：推荐的一端口方案

这是**更适合当前项目真正对外使用**的方案。

### 核心思路

先把前端和后端整理成**同源访问**，再通过 `frp` 只暴露一个端口。

推荐结构如下：

- 本机用 Nginx / Caddy 等**反向代理或静态文件服务器**托管打包后的前端
- API 仍然走本机后端
- 对外只开放一个统一入口，例如本机 `8080`
- ECS `18080` -> 本机 `8080`

注意：

- **当前代码下不能把 FastAPI 直接当成前端静态托管入口使用**。现有 `app/main.py` 只挂载了 `/media`，根路径 `/` 返回的是 JSON，不会直接返回前端页面。

### 外部访问效果

- 用户统一访问：`http://<ECS公网IP>:18080`
- 前端、接口、媒体资源走同一来源
- `CORS` 问题会显著减少
- 但在**当前代码不修改**前提下，如果要完整使用所有功能，仍需要在前端设置里手动填写同一个公网 `backendUrl`
- 这个 `backendUrl` 不是服务器全局配置，而是保存在**各自浏览器的 `localStorage`** 里，所以每个外部用户首次使用都要各自填写一次

### 额外安全提醒

- 本文中的 `http://<ECS公网IP>:...` 示例，适合用于链路验证、部署说明和“当前代码现状”分析，不等于适合长期让外部用户直接填写真实第三方凭证。
- 当前前端会把 `backendUrl`、LLM/Image/Video Key 保存在各自浏览器的 `localStorage` 中，并通过请求头发给后端。
- 因此如果你要让外部用户在浏览器设置页填写真实第三方 API Key，统一入口应优先改成 **HTTPS**；如果当前阶段仍只有 HTTP，更安全的做法是只使用服务端 `.env` 里的默认 Key，不让外部用户自行填写真实 Key。

### 适用条件

这个方案适合当前阶段作为“最稳的无代码公网方案”，但默认还隐含下面几个前提：

- 本机 `8080` 已经正确托管前端打包产物并反向代理后端
- 服务端 `.env` 已准备好你希望对外开放的真实能力所需 Key
- 你接受当前阶段外部用户可能仍需要做少量浏览器本地设置
- 你接受当前仓库仍存在发布前未清理的 Mock 逻辑约束

### 为什么我更推荐这个方案

因为它更符合当前仓库后续“给其他人用”的目标：

- URL 更简单
- 不容易出现前端请求回到用户自己本地 `localhost`
- 浏览器跨域问题更少
- 后续切换到域名和 HTTPS 更顺

### 当前方案要额外满足的代理要求

如果你采用单端口同源方案，统一入口至少还要满足下面几个条件，否则即使 `frp` 打通，功能也可能不完整：

1. **SPA History 回退**
   - 当前前端使用的是 `createWebHistory()`。
   - 需要把 `/step1`、`/step2`、`/step3`、`/step4`、`/video-generation`、`/settings`、`/history` 等前端路由都回退到 `index.html`。
   - 否则用户刷新页面或直接打开子路由时会 404。

2. **SSE / 流式接口不能被代理层缓冲**
   - 当前故事聊天和剧本生成使用 `StreamingResponse` + `text/event-stream`。
   - 如果代理层开启缓冲，前端会出现流式输出卡住、延迟到最后一起显示、甚至超时的问题。
   - 因此统一入口应关闭对应流式接口的代理缓冲，并适当放宽读取超时。

3. **媒体路径必须原样透传**
   - `/media` 必须直接转发到后端。
   - 不能被错误重写到其他静态目录，否则角色图、音频、视频资源会失效。

## 5. 阿里云侧实施方案

## 5.1 ECS 选择建议

建议：

- 系统：Ubuntu 22.04 LTS 或 24.04 LTS
- 公网能力：优先 EIP；如果实例本身已经有公网 IPv4 也可以
- 带宽：先按 3Mbps~10Mbps 试跑，后续根据访问量调整

说明：

- 如果只是临时验证，ECS 自带公网 IP 也够用。
- 如果你希望公网地址更稳定、以后更易迁移，EIP 更灵活。

## 5.2 安全组放行端口

最小建议如下：

| 端口 | 方向 | 用途 | 是否必须 |
|------|------|------|---------|
| `7000/tcp` | 入站 | `frps` 控制连接端口 | 必须 |
| `15173/tcp` | 入站 | 前端公网访问端口（双端口方案） | 视方案而定 |
| `18000/tcp` | 入站 | 后端公网访问端口（双端口方案） | 视方案而定 |
| `18080/tcp` | 入站 | 统一 Web 入口（单端口方案） | 视方案而定 |
| `7500/tcp` | 入站 | `frps` Dashboard（仅管理） | 可选 |

安全建议：

- `7000` 如果本机出口 IP 固定，最好只放行你的出口 IP。
- `7500` 不要对全网开放，最好只允许你自己的管理 IP。
- 业务端口只开放实际用到的那几个，不要图省事全放开。

## 6. FRP 配置设计

## 6.1 ECS 上部署 `frps`

建议目录：

- 程序目录：`/opt/frp`
- 配置目录：`/etc/frp`

示例 `frps.toml`：

```toml
bindPort = 7000

auth.method = "token"
auth.token = "CHANGE_ME_TO_A_STRONG_TOKEN"

transport.tls.force = true

webServer.addr = "127.0.0.1"
webServer.port = 7500
webServer.user = "admin"
webServer.password = "CHANGE_ME_TO_A_STRONG_PASSWORD"
```

说明：

- `bindPort` 是 `frpc` 连到云端的入口端口。
- `auth.token` 必须和客户端一致。
- `transport.tls.force = true` 用来强制 TLS 连接。
- Dashboard 建议只监听 `127.0.0.1`，如果要远程看，优先走 SSH 隧道。

## 6.2 `frps` systemd 服务

`/etc/systemd/system/frps.service`

```ini
[Unit]
Description=frp server
After=network.target

[Service]
Type=simple
ExecStart=/opt/frp/frps -c /etc/frp/frps.toml
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now frps
sudo systemctl status frps
```

## 8. 当前代码状态下的最终判断

这部分是本次核查后的核心结论，按“**当前代码不改动**”来判断。

## 8.1 审计结论

`frp` 本身没有问题，真正决定成败的是**当前仓库的地址回落逻辑**。

已确认的关键事实：

1. `frontend/src/api/story.js` 的大多数故事相关接口，在 `backendUrl` 留空时走相对路径，通常可以配合 Vite 代理或同源入口工作。
2. `frontend/src/views/VideoGeneration.vue` 在开发模式下会把默认后端地址回落到 `http://localhost:8000`。
3. `frontend/src/components/CharacterDesign.vue` 在 `backendUrl` 留空时，媒体地址会回落到 `http://localhost:8000`。
4. `app/main.py` 里的 `CORS` 只允许 `http://localhost:5173`。
5. `app/schemas/pipeline.py` 和 `app/routers/pipeline.py` 中仍保留了 `localhost:8000` 作为部分 `base_url` 默认值。
6. `frontend/src/stores/settings.js` 当前把 `MOCK_ENABLED` 设为 `true`，并且 `useMock` 取决于浏览器本地是否存在 `llmApiKey`。
7. `frontend/src/views/VideoGeneration.vue` 在 `settings.useMock === true` 时，会直接走前端模拟分镜分支，而不是自动复用服务端 `.env` 中的 LLM Key。
8. 图片、视频、LLM 相关能力虽然可以由后端 `.env` 提供默认 Key，但前端设置页里的 `backendUrl`、LLM/Image/Video Key 默认都是浏览器本地存储，不会自动同步给其他外部用户。
9. `frontend/src/views/SettingsView.vue` 的设置页输入框 placeholder 仍提示“留空使用默认 `http://localhost:8000`”。
10. `app/main.py` 根路径 `/` 返回的说明 JSON 里仍写着前端地址 `http://localhost:5173`。

因此，当前仓库的真实结论是：

- **可以做 FRP 公网访问**
- **但不能把“双端口默认开发形态”当成完整公网方案**
- **不改代码时，最稳的做法是单端口同源入口 + 前端手动填写同一个公网 `backendUrl` + 服务端预置所需 `.env` Key**
- **如果公网入口仍是纯 HTTP，则不应要求外部用户在浏览器里填写真实第三方 Key；否则 Key 会落在浏览器本地存储里，并通过明文 HTTP 请求发送**
- **如果目标是“外部用户不在本机、只打开网址就直接完整使用真实功能”，当前代码和当前文档描述仍不能完全保证这一点**

## 8.2 各种方式的可行性矩阵

| 方式 | 是否完整可用 | 是否推荐 | 结论 |
|------|-------------|---------|------|
| 本地开发：`5173 + 8000` | 是 | 是 | 当前本地开发模式正常 |
| FRP 双端口：前端 `15173 -> 5173`，后端 `18000 -> 8000`，`backendUrl` 留空 | 否 | 否 | 部分故事接口可经 Vite 代理使用，但视频页和角色图会误连 localhost |
| FRP 双端口：前端 `15173 -> 5173`，后端 `18000 -> 8000`，`backendUrl` 填公网后端地址 | 否 | 否 | 虽然绕过了 Vite 代理，但会触发 `CORS` 限制 |
| FRP 单端口同源：`18080 -> 8080`，`backendUrl` 留空 | 不稳定 | 不建议直接上 | 视频页大多能按当前来源工作，但角色图仍可能误连 localhost |
| FRP 单端口同源：`18080 -> 8080`，`backendUrl` 手动填 `http://<ECS公网IP>:18080` | 基本可用，但仍有额外前提 | **推荐** | 还需处理浏览器本地设置与 Mock 模式限制，才接近外部完整可用 |

## 8.3 不能直接采用的方案

当前代码下，**不建议**把下面的模式直接给外部用户用：

```text
前端：http://<ECS公网IP>:15173
后端：http://<ECS公网IP>:18000
backendUrl：留空
```

原因：

1. 故事页大多还会走 Vite 代理，看起来像“能用”。
2. 但 `VideoGeneration.vue` 和 `CharacterDesign.vue` 会把部分请求或媒体地址改成 `http://localhost:8000`。
3. 对外部用户来说，这个 `localhost` 指向的是**用户自己的电脑**，不是你的主机。

这会导致：

- 首页可能能打开
- 部分接口可能能用
- 一到角色图、媒体预览、TTS、视频生成、轮询状态等功能就出错或连错

## 8.4 当前最稳的无代码公网方案（不是零配置方案）

推荐你在**不改代码**前提下采用：

```text
外部用户
  |
  v
http://<ECS公网IP>:18080
  |
  v
ECS frps
  |
  v
本机 frpc
  |
  v
本机统一入口 8080
    |- /      -> 前端页面
    |- /api   -> 127.0.0.1:8000
    |- /health-> 127.0.0.1:8000
    |- /media -> 127.0.0.1:8000
```

然后在前端设置页中，手动把 `backendUrl` 填成：

```text
http://<ECS公网IP>:18080
```

同时，如果你不希望外部用户再分别填写 LLM / 图片 / 视频相关 Key，还需要保证服务端 `.env` 已经预置至少一套可用的真实凭证。

这样做的好处：

1. 前后端同源，不会被当前 `CORS` 限制拦住。
2. `VideoGeneration` 与 `CharacterDesign` 不会再回落到用户自己的 localhost。
3. 浏览器只看到一个入口，排查也最简单。

但这里必须补一句非常关键的话：

- 这仍然只是当前仓库下**最稳的无代码方案**，不是“外部用户打开网址即可零配置完整使用”的最终保证方案。
- 如果浏览器没有正确设置 `backendUrl`，或者前端仍处于 `Mock` 模式，或者后端 `.env` 没有准备好所需 API Key，功能仍可能不完整。

## 8.5 当前代码下必须遵守的规则

### 规则 1：双端口方案只能用于链路验证

你可以用双端口方案来确认：

- `frps` 可连接
- `frpc` 正常工作
- ECS 公网端口开放正确

但不要把它直接作为“外部完整使用方案”。

### 规则 2：完整公网使用时，不要依赖 `backendUrl` 留空

当前代码下，如果你要给别人完整使用角色图、视频页、媒体资源，请在前端设置里明确填：

```text
http://<ECS公网IP>:18080
```

并且要注意，这个设置默认保存在当前浏览器本地；换一台电脑、换一个浏览器、或者无痕窗口重新打开时，都要重新检查这项设置是否已经填写。

### 规则 2.1：如果目标是“别人打开网址即用”，必须额外处理浏览器本地 Mock 模式

当前代码里：

- `frontend/src/stores/settings.js` 把 `MOCK_ENABLED` 设为 `true`
- `useMock` 依据的是浏览器本地是否有 `llmApiKey`
- `frontend/src/views/VideoGeneration.vue` 在 `useMock === true` 时会直接生成模拟分镜

这意味着：

- 即使服务端 `.env` 已经配置好默认 LLM Key
- 只要外部用户自己的浏览器里没有本地 `llmApiKey`
- 视频页仍可能进入模拟流程，而不是真实调用

所以按当前代码现实来讲，**“外部用户只打开网址即可完整正常使用真实功能”这一点仍不能完全保证**。

如果你要接近这个目标，至少还要同时满足下面条件之一：

1. 每个外部用户在自己的浏览器设置里填写可用的 `llmApiKey`
2. 后续进入代码修复阶段，把发布环境的 `MOCK_ENABLED` 关闭，并梳理所有本地回落逻辑

### 规则 2.2：如果不希望用户自己填各类 Key，服务端必须预置可用凭证

当前项目的图片、视频、LLM 能力来源分两类：

- 浏览器设置页里由用户自己填写的 Key
- 服务端 `.env` 中的默认 Key

如果你的目标是“外部用户只打开网址即可用”，那就不能依赖外部用户自己逐个填写这些 Key，而要保证服务端 `.env` 至少已经准备好实际要用到的：

- LLM Key
- 图片生成 Key
- 视频生成 Key

否则公网访问虽然能打开页面，但一旦触发对应能力，就会返回未配置或鉴权失败。

### 规则 2.3：如果当前统一入口仍是 HTTP，不要让外部用户在浏览器里填写真实第三方 Key

当前代码里：

- `backendUrl`、LLM/Image/Video Key 默认保存在浏览器本地 `localStorage`
- 这些 Key 会随着请求头一起发给后端

这意味着：

- 如果你当前对外开放的是 `http://<ECS公网IP>:18080`
- 又让外部用户在前端设置页里填写真实第三方 API Key
- 那这些 Key 就会同时暴露在用户自己的浏览器本地存储和明文 HTTP 传输链路里

因此当前阶段应优先遵守下面的次序：

1. 优先只用服务端 `.env` 里的默认 Key，不让外部用户填写真实 Key
2. 如果确实要让外部用户自己填写真实 Key，先把统一入口切到 HTTPS

### 规则 3：最好只给外部暴露一个入口

推荐只给用户一个入口：

```text
http://<ECS公网IP>:18080
```

不要同时让用户直接接触：

- `http://<ECS公网IP>:15173`
- `http://<ECS公网IP>:18000`

除非你只是自己做调试。

## 9.1 第一步：先验证 FRP 链路

只验证网络链路，不验证完整业务。

- 不能证明“当前项目完整公网可用”

## 9.2 第二步：本机准备统一入口

本机需要有一个统一入口，例如 `127.0.0.1:8080`，满足：

- 托管打包后的前端页面（例如 `frontend/dist`）
- `/` 访问前端
- `/api` 转发后端
- `/health` 转发后端
- `/media` 转发后端

这一层可以由本机 Nginx、Caddy 或其他反向代理完成。

补充说明：

- 这里不是直接复用 `app/main.py` 的 `/`
- 也不是只开 `npm run dev` 就算完成
- 如果缺少 `index.html` 回退、SSE 无缓冲、`/media` 原样透传，这一步虽然“有 8080 端口”，但仍不算真正完成

## 9.3 第三步：只通过 FRP 暴露统一入口

推荐只映射：

- ECS `18080 -> 本机 8080`

外部统一使用：

- `http://<ECS公网IP>:18080`

## 9.4 第四步：前端设置中手动填写 `backendUrl`

当前代码下，完整公网使用时建议明确填：

```text
http://<ECS公网IP>:18080
```

补充说明：

- 这是当前**不改代码**前提下的必需步骤，不是可选优化
- 这是浏览器本地设置，不是服务器配置项
- 如果是给多个外部用户使用，需要让每个用户各自在自己的浏览器里完成一次
- 即使服务端 `.env` 已配置默认 LLM Key，这一步也不能替代对前端 Mock 模式的额外确认
- 如果当前统一入口仍是 `http://`，不要要求外部用户在设置页里填写真实第三方 API Key；这些 Key 会保存在各自浏览器 `localStorage`，并通过请求头发送。对外使用时优先改成 HTTPS，或只依赖服务端 `.env` 默认 Key。

## 9.5 第五步：确认是否满足“真实模式”验收口径

如果你的目标不是“能打开页面”，而是“外部用户实际使用真实能力”，上线前还要额外确认：

4. 如果统一入口仍是 HTTP，没有要求外部用户在浏览器里填写真实第三方 Key
5. 你对外宣称的能力范围，与当前实际可用能力一致

如果上面 5 点做不到，就不要把当前方案描述成“打开网址即可完整使用”。

## 11. 风险、限制与最终建议

## 11.1 当前文档能保证什么

本文能保证的是：

- 把当前仓库和 FRP 公网访问相关的关键实现点讲清楚
- 指出哪些模式在当前代码下会连错或冲突
- 给出在**不改代码**前提下最稳的公网使用方式

## 11.2 当前文档不能替代代码修复

由于你明确要求“现在不改代码”，必须如实说明：

- 当前仓库里仍然保留若干 `localhost` 回落逻辑
- 当前后端仍然保留双端口场景下的 `CORS` 限制
- 当前设置页 placeholder 和 API 根路径说明里也仍保留 `localhost` 开发提示，公网联调时容易误导排障
- 当前前端的 `backendUrl` 以及多类 API Key 默认保存在浏览器本地，不会随着 FRP 部署自动同步给其他用户
- 如果公网入口仍是纯 HTTP，上述浏览器本地 API Key 方案不适合让外部用户填写真实第三方凭证
- 当前前端仍存在 `MOCK_ENABLED = true` 的发布前限制，至少视频页不能自动保证走真实链路
- 这些问题不会因为只上 `frp` 自动消失

## 11.3 最终建议

当前阶段的正确结论是：

1. **FRP 可行**
2. **双端口方案只适合链路验证**
3. **不改代码时，最稳的公网方式是“单端口同源入口 + 手动设置同一公网 `backendUrl` + 服务端预置所需 `.env` Key”**
4. **如果你的验收标准只是“先让公网能访问”，做到第 3 条已经足够**
5. **如果你的验收标准是“外部用户打开网址即可直接正常使用真实功能”，当前代码和当前文档仍不能完全保证，需要额外处理浏览器本地 Mock 模式、默认 API Key 策略，以及纯 HTTP 入口下不应收集用户真实第三方 Key 的限制**
6. 如果后续要彻底消除这些限制，再进入代码修复阶段
