# Script Generation Concurrency
> 生成日期：2026-04-03

## 背景

`/api/v1/story/generate-script` 会按集生成剧本层内容。
当前实现支持在单次请求内部并发生成多集，以缩短整批 6 集剧本的总耗时，同时保持前端接口不变。

从 2026-04-03 起，剧本生成并发数不再写死在代码里，而是改为后端配置项。

## 配置项

后端新增配置项：

- `script_generation_concurrency`

对应环境变量：

```env
SCRIPT_GENERATION_CONCURRENCY=3
```

默认值：

- `3`

取值范围：

- 最小 `1`
- 最大 `6`

如果配置小于 `1`，会回退到 `1`。
如果配置大于 `6`，会被钳制到 `6`。

## 运行行为

单次 `generate-script` 请求的实际并发数按以下规则计算：

```text
min(script_generation_concurrency, 当前待生成的集数)
```

例如：

- 配置为 `3`，待生成 6 集时，内部并发为 `3`
- 配置为 `3`，只续生成第 5-6 集时，内部并发为 `2`
- 配置为 `1`，则退化为串行生成

## 当前实现特点

### 1. 仅控制单次请求内部并发

这个配置项只控制一次 `/generate-script` 调用内部同时发起多少个 LLM 请求。
它不是全局限流，也不是跨请求总并发上限。

### 2. 输出顺序仍按集数稳定

即使后端内部并发执行，返回给前端和落库时仍然会按 `outline` 顺序输出，避免第 3 集先于第 2 集显示。

### 3. 接口形态不变

前端仍然调用原有 `/api/v1/story/generate-script` SSE 接口，不需要修改请求参数。

## 调优建议

- `1`：最保守，适合排查 provider 稳定性问题
- `2`：更稳健，适合限流较严或超时较多的 provider
- `3`：当前默认值，适合作为日常运行配置
- `4-6`：总耗时可能继续下降，但更容易触发 provider 侧限流、超时或瞬时资源压力

建议优先从 `3` 开始观察，如果遥测中出现明显的超时、429、连接中断或 JSON 不完整，再回调到 `2`。

## 修改方式

在后端 `.env` 中设置：

```env
SCRIPT_GENERATION_CONCURRENCY=3
```

然后重启后端服务，使配置生效。

## 回退方式

如果需要快速回退为串行生成，可设置：

```env
SCRIPT_GENERATION_CONCURRENCY=1
```
